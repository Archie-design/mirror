## 二、驗證方式說明

本份報告採取兩階段驗證：

| 階段 | 範圍 | 由誰執行 | 已完成？ |
|------|------|---------|:-:|
| 程式碼層審查 | 確認功能存在、編譯通過、按鈕對應到正確流程、權限守衛無漏 | 開發團隊 | ✅ |
| 實機操作驗證 | 實際登入、打卡、提交、審核，確認資料正確寫入資料庫 | 營運團隊 | ⏳ 上線前 |

**程式碼層審查通過 ≠ 上線即可運作**：仍需在 production 環境跑一輪實機操作，特別是依賴排程觸發與外部服務（LINE Login）的功能。

## 三、功能驗收清單 — 學員主流程

### I-1. 每日打卡（每日踏程分頁）

#### 1.1 基本任務（d1 ～ d8，共 8 項）

- **位置**：`每日踏程` > 上方「基本」區塊
- **預期行為**：學員每日可從 8 項基本定課中任選 3 項打卡，每項加 20 分；打卡後該按鈕轉為「已完成 + 完成時間」並可點擊撤回
- **驗收狀態**：✅
- **驗證方法**：登入學員帳號 → 進入「每日踏程」 → 點擊任一基本任務（如「五感恩」） → 應立刻顯示綠色已完成狀態與時間 → 重新整理頁面後仍存在
- **備註**：每日上限 3 次（含撤回後再打）；超過上限時按鈕變灰

#### 1.2 加權任務（p1 ～ p5，共 5 項）

- **位置**：`每日踏程` > 「加權」區塊
- **預期行為**：每日可從 5 項加權定課（打拳、觀心書、大悲咒、子時入睡、痛參）任選 3 項打卡，每項加 50 分
- **驗收狀態**：✅
- **驗證方法**：同上，點擊任一加權任務（如「打拳」） → 顯示已完成 + 時間
- **備註**：與基本任務獨立計算，互不影響上限

#### 1.3 破曉打拳（p1_dawn）

- **位置**：`每日踏程` > 「加權」區塊內、p1 打卡後出現
- **預期行為**：當日已完成「打拳（p1）」且**台灣時間早於 12:00** 時出現紅色「破曉打拳」按鈕，加 50 分加成
- **驗收狀態**：✅
- **驗證方法**：請於上午 11 點前先打卡 p1，再點擊破曉打拳；中午過後應自動隱藏該按鈕（每分鐘自動更新）
- **備註**：規則同時在前端 UI 與後端資料庫雙重檢核，下午硬呼叫亦會被拒絕

#### 1.4 飲控任務（diet_veg / diet_seafood）

- **位置**：`每日踏程` > 「飲控」區塊
- **預期行為**：每日二選一 — 「三餐吃素」+50 分，或「三餐海鮮素」+30 分；選定後另一項禁用
- **驗收狀態**：✅
- **驗證方法**：點擊「三餐吃素」 → 該項顯示已完成，「三餐海鮮素」按鈕禁用
- **備註**：撤回後可重新選擇

#### 1.5 任務收藏（★ 常用標記）

- **位置**：`每日踏程` > 上方「編輯」按鈕
- **預期行為**：點擊「編輯」進入收藏模式，每項任務出現★星號，點擊切換收藏狀態；常用任務優先顯示
- **驗收狀態**：✅
- **驗證方法**：點擊「編輯」 → 為「五感恩」標記★ → 退出編輯模式 → 確認該任務排在前列
- **備註**：收藏記錄存於使用者本機瀏覽器（換裝置不會同步）

### I-2. 旅伴週報（週任務）

#### 2.1 主題週期顯示

- **位置**：`旅伴週報` > 頁首
- **預期行為**：依當前週次顯示對應的旅程主題（含主題名稱、第幾週、敘事描述）
- **驗收狀態**：✅
- **驗證方法**：進入「旅伴週報」 → 頁首應顯示主題卡片
- **備註**：主題對應規格 §1.3「四階段」與 §3.4「主題對應」

#### 2.2 破框練習（wk1）

- **位置**：`旅伴週報` > 「破框練習」區塊
- **預期行為**：每週最多 3 次、每次加 200 分；提供週一～週日七個按鈕，可選日打卡
- **驗收狀態**：✅
- **驗證方法**：選擇本週任一日按鈕 → 該日按鈕轉為已完成
- **備註**：跨週後本週上限重新計算

#### 2.3 天使通話（wk2）

- **位置**：`旅伴週報` > 「天使通話」區塊
- **預期行為**：每週最多 2 次、每次加 200 分
- **驗收狀態**：✅
- **驗證方法**：同破框練習操作模式
- **備註**：滿 2 次後本週按鈕全部禁用

#### 2.4 小組凝聚 — 線上（wk3_online）

- **位置**：`旅伴週報` > 「小組凝聚（線上）」區塊
- **預期行為**：每週 1 次、加 100 分；學員填寫補充說明 → 送審 → 小隊長一級審核 → 通過即直接入帳
- **驗收狀態**：✅
- **驗證方法**：填寫表單 → 送出後狀態變「審核中」 → 切換小隊長帳號至「隊長基地」核准 → 學員端應變為「已完成」
- **備註**：退回後本週可重新提交

#### 2.5 小組凝聚 — 實體（wk3_offline）

- **位置**：`旅伴週報` > 「小組凝聚（實體）」區塊
- **預期行為**：每週 1 次、基礎加 300 分（全員到齊再加 100 分、大隊長到場再加 100 分）；流程：大隊長排定日期 → 當日掃 QR 報到 → 小隊長送審 → 大隊長終審 → 全員入帳
- **驗收狀態**：⚠️
- **驗證方法**：① 大隊長端排定日期 → ② 當日由小隊長掃描現場 QR → ③ 隊員逐一掃碼報到 → ④ 小隊長送審 → ⑤ 大隊長終審 → 全員應加分
- **備註**：完整流程需多角色協作，建議上線前先用測試帳號跑一次完整循環

#### 2.6 人生大戲分享（wk4_small / wk4_large）

- **位置**：`人生大戲` > 「分享打卡」區塊
- **預期行為**：小群分享每週 1 次加 200 分；大群分享每週 1 次加 300 分。**前提**：本週至少完成 1 個九宮格
- **驗收狀態**：✅
- **驗證方法**：本週尚未打過任一格 → 分享按鈕應禁用 → 完成任一格後 → 按鈕解鎖
- **備註**：未達條件時系統自動阻擋，避免誤打卡

#### 2.7 臨時加碼任務

- **位置**：`旅伴週報` > 最下方「官方加碼」區塊
- **預期行為**：管理員後台建立的限時任務動態出現；每項固定每週 1 次
- **驗收狀態**：✅
- **驗證方法**：請管理員至後台新增測試任務 → 重新整理「旅伴週報」 → 應出現該任務
- **備註**：點擊即直接入帳，無須審核

### I-3. 人生大戲（九宮格）

#### 3.1 五運自評滑桿

- **位置**：`人生大戲`（首次進入）
- **預期行為**：五項滑桿（事業/財富/情感/家庭/體能運）各 1～10 分，提交後自動配對最弱的一項對應「旅伴」
- **驗收狀態**：✅
- **驗證方法**：拖動滑桿 → 點「完成評分」 → 應跳轉至「找到我的旅伴」結果頁
- **備註**：分數寫入 `CharacterStats` 五運欄位，僅本人可改

#### 3.2 旅伴配對與九宮格初始化

- **位置**：`人生大戲` > 旅伴配對結果頁
- **預期行為**：顯示配對到的旅伴卡片（小桃/小草/小鐵/小獅/翡翠城）並自動套用對應的九宮格公版模板
- **驗收狀態**：✅
- **驗證方法**：完成五運評分後 → 應顯示旅伴卡片與 9 個未完成格子
- **備註**：旅伴可重選，但會重置九宮格進度

#### 3.3 九宮格格子打卡

- **位置**：`人生大戲` > 9×9 網格
- **預期行為**：點擊任一格 → 標示為已完成 + 完成時間。每格本身不計分；僅在連成一線時觸發 +300 分
- **驗收狀態**：✅
- **驗證方法**：依序點擊 3 個對齊的格子（橫/直/斜） → 應跳出「+300 分」提示
- **備註**：連線上限 8 條（最高 +2400 分）

#### 3.4 旅伴卡片詳情

- **位置**：`人生大戲` > 上方旅伴標籤
- **預期行為**：點擊旅伴標籤展開卡片，顯示原型、象徵、故事意涵
- **驗收狀態**：✅
- **驗證方法**：點擊旅伴名稱 → 應展開描述文字
- **備註**：對應規格 §1.1 旅伴系統

### I-4. 旅人榜（排行榜）

#### 4.1 期間切換（週/月/累積）

- **位置**：`旅人榜` > 上方期間按鈕列
- **預期行為**：三個期間按鈕，點擊切換顯示對應期間的分數
- **驗收狀態**：✅
- **驗證方法**：點擊「週榜」 → 分數與排序變化；點「月榜」 → 再次變化；「累積」 → 顯示總分
- **備註**：本週/本月即時計算；歷史週/月需 production 環境累積快照後才有資料

#### 4.2 範圍切換（個人/小隊/大隊）

- **位置**：`旅人榜` > 期間下方範圍按鈕列
- **預期行為**：個人榜顯示 Top 100 學員；小隊榜顯示各小隊平均分；大隊榜顯示各大隊平均分
- **驗收狀態**：✅
- **驗證方法**：切換三個按鈕應看到不同維度的排序
- **備註**：累積榜的小隊聚合會將大隊長計入大隊內每個小隊（依規格 §6）

#### 4.3 歷史期間下拉選單

- **位置**：`旅人榜` > 切到週/月榜時出現的下拉選單
- **預期行為**：可查詢過去 12 週/月的歷史排行
- **驗收狀態**：⚠️
- **驗證方法**：上線後等首次自動快照產生（週日 16:30 UTC 後） → 下拉選單應出現新選項
- **備註**：尚未產生快照前下拉選單為空，屬正常現象

#### 4.4 統計區間標籤

- **位置**：`旅人榜` > 期間選單旁
- **預期行為**：顯示當前查看的具體日期區間（例「2026 年 5/4 ~ 5/10」）
- **驗收狀態**：✅
- **驗證方法**：切換不同期間時應自動更新
- **備註**：累積榜顯示「活動全期」

#### 4.5 個人榜細項展開（RBAC）

- **位置**：`旅人榜` > 小隊榜上點擊隊伍列
- **預期行為**：依角色限制展開細項 — 學員只能展開自己所在小隊；小隊長展開本隊全員；大隊長展開本大隊任一小隊；GM 展開全部
- **驗收狀態**：✅
- **驗證方法**：以學員帳號登入，嘗試展開其他小隊 → 應顯示無權查看
- **備註**：本期已修正大隊長無法展開「零得分小隊」的邏輯漏洞

### I-5. 我的旅程（個人統計與一次性任務）

#### 5.1 累積積分卡

- **位置**：`我的旅程` > 頁首
- **預期行為**：顯示當前總積分（千位分隔）與連續打卡天數
- **驗收狀態**：✅
- **驗證方法**：打卡後應立即更新分數
- **備註**：連續打卡天數依「邏輯日」計算（中午 12:00 切換）

#### 5.2 一次性任務 o1（傳愛截止 7/11）

- **位置**：`我的旅程` > 「一次性任務」區塊
- **預期行為**：學員可提交 1 次申請（訪談對象 + 完成日期 + 備註）；流程：學員提交 → 小隊長初審 → 大隊長終審 → 入帳
- **驗收狀態**：✅
- **驗證方法**：填寫表單 → 送出 → 狀態變「審核中」
- **備註**：截止日 7/11；同一申請只能提交 1 次

#### 5.3 一次性任務 o2_1 ～ o2_4（戲劇進修，各 1 次）

- **位置**：`我的旅程` > 「戲劇進修」四個子項目
- **預期行為**：四個子項目各可提交 1 次；屬「一級審核」 — 小隊長核准後直接入帳
- **驗收狀態**：✅
- **驗證方法**：各別提交 → 小隊長核准 → 立即加分（無大隊長環節）
- **備註**：截止日 7/1

#### 5.4 一次性任務 o3, o4（各 1 次，二級審核）

- **位置**：`我的旅程` > 對應區塊
- **預期行為**：學員提交 → 小隊長初審 → 大隊長終審 → 入帳
- **驗收狀態**：✅
- **驗證方法**：依次經過兩級審核應正確加分
- **備註**：截止日 7/1

#### 5.5 一次性任務 o5, o6, o7（可多次提交，二級審核）

- **位置**：`我的旅程` > 對應區塊
- **預期行為**：可重複提交多份（每位被介紹人/課程一筆）；同樣需兩級審核
- **驗收狀態**：✅
- **驗證方法**：連續提交三筆 → 應全部進入待審清單
- **備註**：o5, o6 截止 7/1；o7 截止 7/11

#### 5.6 申請狀態追蹤

- **位置**：`我的旅程` > 各任務區塊內已提交清單
- **預期行為**：每筆申請顯示：審核中 / 初審通過 / 已核准 / 未通過 + 駁回原因
- **驗收狀態**：✅
- **驗證方法**：提交後依次經過審核流程，狀態應同步變化
- **備註**：駁回後可在截止日前重新提交

#### 6.1 場次列表

### VI-1. 每週快照（weekly-snapshot）★ 本期新增

- **觸發時機**：每週日 16:30 UTC = 台灣時間週一 00:30

- **備註**：可手動觸發 — 由開發團隊以 `curl` 加 `CRON_SECRET` 標頭呼叫

### VI-2. 每月快照（monthly-snapshot）★ 本期新增

- **觸發時機**：每月 1 號 16:30 UTC = 台灣時間 1 號 00:30
- **功能**：將上個月每位學員的當月累積分快照入庫，供「上月榜」使用
- **驗收狀態**：⚠️
- **驗證方法**：每月 1 號 01:00 後檢查 `MonthlyRankSnapshot`
- **備註**：同上可手動觸發測試

## 九、功能驗收清單 — 後台計分核心邏輯

> 學員看不到但會體驗到的規則。

### VII-1. 邏輯日（中午 12:00 切換）

- **預期行為**：上午 11:59 打卡與下午 12:01 打卡視為同一邏輯日；連續打卡天數依此計算
- **驗收狀態**：✅
- **驗證方法**：請於上午先打卡 → 下午同一天再次打 → 應提示「今日已完成」
- **備註**：對應規格 §5.6

### VII-2. 任務每日上限自動 Cap

- **預期行為**：基本任務超過 3 次/日 → 第 4 次按鈕變灰；加權任務同理
- **驗收狀態**：✅
- **驗證方法**：連打 3 個基本任務 → 第 4 個應禁用
- **備註**：前後端雙重檢核

### VII-3. 連線加分自動觸發

- **預期行為**：九宮格完成第三格構成連線時，立即額外加 300 分
- **驗收狀態**：✅
- **驗證方法**：刻意對齊三格打卡 → 應跳出加分提示
- **備註**：8 條連線後不再加分（最高 +2400 分）

### VII-4. 重複打卡防呆

- **預期行為**：同一邏輯日、同一任務不會被計入兩次
- **驗收狀態**：✅
- **驗證方法**：清除快取後再點同一任務 → 應顯示已完成
- **備註**：以「邏輯日 + 任務 ID」做去重

### VII-5. 累積分數即時更新

- **預期行為**：打卡後 1 秒內個人總分更新；排行榜輪詢時同步
- **驗收狀態**：✅
- **驗證方法**：打卡後留意右上角分數
- **備註**：使用單一資料來源（CharacterStats.Score），確保各畫面一致

### VII-6. 角色權限守衛

- **預期行為**：學員點不到審核按鈕、隊長點不到大隊長審核、大隊長點不到管理後台
- **驗收狀態**：✅
- **驗證方法**：以學員帳號嘗試直接訪問 `/admin` URL → 應跳轉密碼頁
- **備註**：所有後端 server action 都有對應守衛

### VII-7. 管理操作日誌

- **預期行為**：所有管理者操作（轉隊、刪除、改獎勵）皆記錄時間/操作者/受影響對象
- **驗收狀態**：✅
- **驗證方法**：管理後台執行任一變更 → 進入「審核管理 III」應有對應 log
- **備註**：日誌不可刪除

## 十、已知限制與設計取捨

### 10.1 上線前必須執行的資料庫指令稿

下列三支於本期新增，**尚未在 production 執行**：

| 指令稿 | 用途 | 不執行的後果 |
|--------|------|--------------|
| `202605010001_rank_snapshots.sql` | 建立週/月快照表 + 聚合函式 | 排行榜歷史與成長曲線無法使用 |
| `202605010002_uncomplete_cell_rpc.sql` | 九宮格回溯原子化 | 小隊長按下回溯按鈕會出錯 |
| `202605010003_distinct_rank_helpers.sql` | 排行榜歷史下拉選單最佳化 | 歷史下拉選單載入緩慢且可能截斷 |

請於 Supabase SQL Editor 依序執行（已確認可重複執行不會出錯）。

### 10.2 排行榜歷史資料需累積

- **首次觸發時間**：上線後第一個週日 16:30 UTC
- **完整可用時間**：上線後第二週週一起，「上週榜」開始有資料
- **成長曲線**：累積至少 1 週快照後才有第一個資料點，至少 4 週後曲線才較具參考價值

### 10.3 沒有自動測試框架

本系統目前無 Jest/Vitest 等自動化測試。所有功能變更均需人工於瀏覽器驗證。建議：
- 上線前由 2-3 位營運成員以不同角色（學員/隊長/大隊長）各跑一遍流程
- 重點驗證跨角色協作（凝聚審核、一次性任務兩級審核）

### 10.4 LINE Login 服務依賴

LINE Login 整合需 LINE Developers Console 設定正確的 callback URL 與環境變數（`LINE_LOGIN_CHANNEL_ID`、`LINE_LOGIN_CHANNEL_SECRET`）。設定錯誤時學員只能改用「姓名 + 末三碼」登入，不影響其他功能運作。

### 10.5 學員端本機儲存項目

下列資訊存於學員自己的瀏覽器（換裝置/清除快取會遺失）：
- 任務收藏（★ 標記）
- 慶典報名識別碼

不影響核心打卡資料，但建議在學員手冊中說明。
